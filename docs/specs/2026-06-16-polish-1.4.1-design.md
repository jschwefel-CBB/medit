# medit 1.4.1 — Polish + Universal Binary (Design)

**Date:** 2026-06-16
**Status:** Approved (decisions captured via brainstorming)
**Version target:** 1.4.1

## Goal

A small, low-risk polish release plus a reach improvement, with no new entitlements:

1. **Empty-document gutter** — hide the line-number ruler whenever the editor text is empty; show it the moment there's content.
2. **Toggle audit** — surface every reasonable behavior as a user toggle: add 6 smart-substitution editor toggles, fix the `syncSidebarWithActiveTab` orphan, fill menu/Settings parity gaps (status bar, invisibles), and add editor padding as a numeric setting.
3. **Status-bar wrap control** — a clickable "Wrap: On/Off" segment in the status bar that toggles word wrap.
4. **Universal Intel+ARM binary** — ship an `x86_64 + arm64` app so it runs natively on both architectures.

**Out of scope (deferred):** rainbow-depth bracket highlighting (moved to 1.5), notarization (deferred until the Apple Developer account is verified).

## Tech stack

Swift 6 / AppKit, local SwiftPM package `MeditKit` + thin Xcode app target. No new dependencies. TDD for all pure/preference logic; AppKit pieces verified via the existing smoke-test pattern and live.

## 1. Empty-document gutter

### Behavior
The line-number ruler is visible only when **both** `prefs.showLineNumbers` is true **and** the editor's text is non-empty. Transitions:

| State | Gutter |
|---|---|
| Untitled, empty (`""`) | hidden |
| Untitled, user types first char | appears |
| User deletes all text back to empty | hidden again |
| File opened with content | shown |
| File opened but empty on disk | hidden |
| `showLineNumbers` pref off | hidden (regardless of text) |

This keys on **empty text**, not untitled/no-file (per the decision). The app is always backed by an `NSDocument`; the "no document" launch window is an untitled doc with `text == ""`.

### Mechanism
`EditorViewController.configureRuler(visible:)` is the single choke point. Change it so the *effective* visibility folds in an emptiness check, and call it from the places that observe text changes.

- Use the `scrollView.rulersVisible = false` / `hasVerticalRuler = false` path to hide — **not** `ruleThickness = 0`, because `LineNumberRulerView.adjustWidth(forLineCount:)` re-inflates thickness to the 2-digit minimum on every draw (`LineNumberRulerView.swift:147,163-170`).
- The governing pref (`prefs.showLineNumbers`) is still the user's intent; emptiness only *additionally* suppresses the gutter.

### Risk
None notable. The ruler instance is retained across hide/show (already the case), so toggling on the empty↔non-empty boundary just flips scroll-view flags — cheap, no flicker. Render-regression guard (editor still shows text) remains green because the text view itself is untouched.

## 2. Toggle audit

### 2a. Six editor smart-behavior toggles
Five are hardcoded `false` in `EditorViewController.loadView()` (lines 95-99) plus `smartInsertDeleteEnabled`; the sixth (continuous spell-check) is currently left at the NSTextView default. Promote all to preferences, each **defaulting OFF** (gedit-like, plain-text-editor behavior — also avoids surprising users who upgrade):

| Pref | NSTextView property | Default |
|---|---|---|
| `smartQuotes` | `isAutomaticQuoteSubstitutionEnabled` | false |
| `smartDashes` | `isAutomaticDashSubstitutionEnabled` | false |
| `automaticTextReplacement` | `isAutomaticTextReplacementEnabled` | false |
| `automaticSpellingCorrection` | `isAutomaticSpellingCorrectionEnabled` | false |
| `smartInsertDelete` | `smartInsertDeleteEnabled` | false |
| `continuousSpellChecking` | `isContinuousSpellCheckingEnabled` | false |

Six total. The first five are the smart-substitution flags that were hardcoded; the sixth, "Check spelling while typing" (the red squiggles), is a genuinely useful editor toggle, default off to match gedit. Surfaced as **Settings checkboxes** in a new "Smart Substitutions" group. Applied both at construction (`loadView`) and live in `preferencesChanged()`. Note: `isContinuousSpellCheckingEnabled` must be set on the text view at construction explicitly (it's not in the current hardcoded block).

### 2b. Fix the `syncSidebarWithActiveTab` orphan
`syncSidebarWithActiveTab` (default true) is registered and has a property, and the View menu's "Reveal Active File in Sidebar" item toggles it via `EditorWindowController.toggleRevealActiveFile`. It is **not** an orphan in behavior, but it has **no Settings checkbox**, unlike the other sidebar toggles. Add a Settings checkbox ("Sidebar: reveal the active file") so it has parity with the menu and the other sidebar prefs. Verify the menu item's checkmark (`validateMenuItem`) reflects the pref.

### 2c. Menu/Settings parity gaps
- `showStatusBar` — menu-only → add Settings checkbox ("Show status bar").
- `showInvisibles` — menu-only → add Settings checkbox ("Show invisibles").

Both already apply live via `preferencesChanged()` (`applyStatusBarVisibility`, `applyShowInvisibles`), so only the Settings UI is missing.

### 2d. Editor padding as a numeric setting
The fixed `textContainerInset = NSSize(width: 4, height: 4)` (`EditorViewController.swift:103`) becomes a preference `editorPadding: Int` (points), default **4**, range 0–40. Surfaced as a numeric field in Settings ("Text padding:"). Applied at construction and in `preferencesChanged()` (set `textView.textContainerInset = NSSize(width: p, height: p)` then trigger a redisplay).

### Settings window reorganization
The Settings window currently lists ~15 controls in one flat column inside a scroll view. Adding ~7 more makes a flat list unwieldy. Introduce lightweight **section headers** (bold labels) to group: **Editor**, **Smart Substitutions**, **Indentation**, **Files**, **Sidebar**. Keep the existing scroll-view + flipped-document-view + top-down Auto Layout chain; headers are just bold `NSTextField` rows pinned in the chain. The last control still pins `content.bottomAnchor`.

### Toggle mechanics (the established pattern)
Per new boolean toggle:
- **Preferences.swift**: `Key` entry; `registerDefaults` entry; `Bool` property with `didChange()` setter.
- **PreferencesWindowController.swift**: stored `NSButton!`; create in `buildUI()` with `checkboxWithTitle:target:action:`; add to the `addSubview` list; add layout constraints chaining under the previous row; write-back line in the relevant action handler (`checkChanged` for editor, `sidebarCheckChanged` for sidebar, or a new `smartSubstChanged`); initial state in `syncFromPrefs()`.
- **EditorViewController.swift** (for editor-affecting ones): read at construction (`loadView`) and re-apply in `preferencesChanged()`.

No new **View-menu** items are required for 2a/2d (they're editor prefs, Settings-appropriate). 2c only *adds* Settings parity for existing menu items; the menu items stay.

## 3. Status-bar wrap control

### Behavior
A clickable inline segment in the status bar reading **"Wrap: On"** / **"Wrap: Off"**, placed in the existing right-side group (after the line-ending button, before the INS/OVR mode label). Clicking it flips `prefs.wrapLines` and applies wrap immediately. It reflects the current wrap state (updates when wrap changes from the menu, Settings, or this control).

### Mechanism
- **StatusBarView.swift**: add a `wrapButton` via the existing `makeInlineButton()` idiom; add a callback `var onWrapToggle: (() -> Void)?`; insert it into the stack; extend `update(...)` to take a `wrap: Bool` and set the title. Follows the `languageButton`/`encodingButton` pattern exactly.
- **EditorViewController.swift**: in the status-bar wiring (where `onLanguagePick`/`onReinterpret` are set), set `statusBar.onWrapToggle = { [weak self] in self?.toggleWrapFromStatusBar() }`. That helper flips `prefs.wrapLines` and calls `applyWrapMode`. Extend the `statusBar.update(...)` call in `updateStatusBar()` to pass `wrap: prefs.wrapLines`.
- Because `applyWrapMode` is already called from `preferencesChanged()` and `toggleWordWrap`, and `updateStatusBar()` runs on selection/text change, the label stays in sync. Add an explicit `updateStatusBar()` after a wrap toggle so the label flips immediately even without a selection change.

### Risk
Low. Pure additive UI in a "dumb display" view. The wrap behavior path (`applyWrapMode`) is unchanged and already shipping.

## 4. Universal Intel+ARM binary

### Behavior
The released `.app` contains both `x86_64` and `arm64` slices, so it runs natively on Intel and Apple-silicon Macs (no Rosetta on Intel-targeted code; the current build is arm64-only).

### Mechanism
- Build the Xcode `medit` scheme for both architectures. The cleanest approach for an Xcode-target app is to set `ARCHS = "x86_64 arm64"` and `ONLY_ACTIVE_ARCH = NO` for the Release configuration (a build-settings change in `project.pbxproj`, or passed on the `xcodebuild` command line for the release build).
- Dependencies: `HighlighterSwift` is a SwiftPM source dependency, so it compiles for both arches automatically — no prebuilt-binary lipo needed.
- Verify with `lipo -archs` / `file` on the built binary that both slices are present before packaging.
- The release zip + signing flow is otherwise unchanged (ad-hoc signed until notarization lands).

### CI note
The existing CI (`swift build`/`swift test` on `macos-14`) validates the SwiftPM package on the runner's native arch. Universal-binary verification is a **release-time** concern (the `.app` is built via `xcodebuild`, not SwiftPM). Optionally add a release build sanity check, but CI's job remains build+test correctness. No CI change is required for this release; the universal flags live in the app target / release build.

## Testing

- **Preferences** (TDD): every new boolean defaults correctly and persists across a reloaded `Preferences`; `editorPadding` defaults to 4, clamps to 0–40. Extend `PreferencesTests`.
- **Gutter logic**: the emptiness gate is in `configureRuler`; add a smoke test that an editor with empty text reports the ruler hidden and a non-empty editor reports it visible (using the existing AppKit smoke-test harness — load the view, set text, assert `scrollView.rulersVisible`).
- **Status bar**: smoke-test that `StatusBarView.update(..., wrap:)` sets the button title, and that `onWrapToggle` fires on click (or at least is wired). The wrap *behavior* is already covered by existing wrap tests.
- **Universal binary**: a manual/release verification step (`lipo -archs`), not a unit test.
- **Regression**: the editor-renders-text smoke test stays green (gutter changes don't touch the text view).
